# Karpathy Guidelines - OpenHuman项目定制版

## 核心原则
基于Andrej Karpathy的AI编程行为准则，减少常见的LLM编码错误。

**权衡**: 这些准则偏向谨慎而非速度。对于简单任务，请使用判断力。

### 核心规则
优先考虑正确性、清晰度和有界变更，而不是速度、推测或过度工程。

## 1. 先想再写
不要假设，不要隐藏困惑，公开权衡。

在实施之前：
- 明确陈述假设
- 如果不确定，询问而不是猜测
- 如果存在多种解释，展示它们——不要默默地选择
- 如果存在更简单的方法，请提出
- 当请求的方法增加不必要的复杂性时，提出反对
- 在任务边界足够清晰可以安全行动之前，不要开始编码

**OpenHuman项目应用**:
- 在OpenHuman项目中，明确碳硅基共生架构假设
- 对于Athena工作流变更，先分析影响范围
- MAREF工作流修改需经过河图策略验证

## 2. 简洁第一
解决所需问题的最少代码，没有推测性内容。

- 不要添加超出请求的功能
- 不要为单次使用代码构建抽象
- 除非被请求，否则不要添加"灵活性"、"可配置性"或"可扩展性"
- 不要为不可能的场景添加错误处理
- 如果你写了200行代码，但可以用50行完成，重写它
- 问自己："高级工程师会说这太复杂吗？"如果是，请简化

**OpenHuman项目应用**:
- Athena工作流代码保持最小化，避免过度工程化
- 基因管理系统遵循单一职责原则
- 保持代码逻辑清晰简洁，便于团队协作

## 3. 精准执行
只修改必须修改的内容。只清理自己造成的混乱。

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式
- 不要重构没有问题的东西
- 匹配现有风格，即使你会用不同的方式
- 如果注意到无关的死代码，提出来——不要删除它

当你的更改创建了孤儿代码时：
- 移除你的更改导致的未使用的导入/变量/函数
- 除非被要求，否则不要移除已有的死代码

测试标准：每个更改的行都应该直接追溯到用户的请求。

**OpenHuman项目应用**:
- 只修改与当前任务相关的Athena Agent代码
- 保持变更精准，避免影响其他工作流
- 匹配项目的代码风格和约定

## 4. 目标驱动执行
定义成功标准。循环直到验证。

- 将命令式任务转化为可验证的目标
- 描述期望的最终状态，而不仅仅是步骤
- 定义清晰的验证步骤（测试、输出、行为）
- 每次变更后，验证向目标的进展
- 如果卡住，解释是什么阻碍了并建议下一步
- 当成功标准满足时停止

Andrej的关键见解：
"LLMs在循环直到满足特定目标方面非常出色...不要告诉它要做什么，给它成功标准然后看着它执行。"

**OpenHuman项目应用**:
- 每个基因管理任务必须有明确的验证标准
- 将复杂任务分解为可验证的小步骤
- 每一步都有明确的完成标准

## 项目特定指导

### OpenHuman项目约定
1. **架构一致性**: 遵循碳硅基共生架构设计原则
2. **测试验证**: 遵循Athena统一测试记录模板
3. **工作流稳定**: 保持Athena工作流的稳定性和可验证性
4. **代码审查**: 所有变更需经过代码审查流程

### MAREF工作流规则
1. **河图策略**: 所有MAREF工作流变更需经过河图策略验证
2. **智能编排**: 保持工作流的智能化和自动化特性
3. **状态管理**: 确保工作流状态的可追踪性

### Athena系统集成
1. **API兼容**: 保持与Athena API的向后兼容性
2. **配置管理**: 使用统一的配置管理系统
3. **监控告警**: 集成到现有的监控告警体系

### LLM Wiki 知识协议

**知识查找优先级**：
```
wiki/ → claude-mem搜索 → memory/日志（解释：先查已蒸馏的结构化知识，不够再搜原始记忆，最后翻日志）
```

第一次遇到不熟悉的概念、架构或代码时：
1. 先读 `wiki/INDEX.md`，通过语义查找表定位相关页面
2. 读目标 wiki 页面，跟踪 `[[wikilinks]]` 获取更多上下文
3. 如果 wiki 不够，用 claude-mem 搜索相关历史
4. 如果还查不到，再读原始源码或日志

**会话结束时自动蒸馏**：
- 复杂工作完成后，对照 `wiki/SCHEMA.md` 检查是否有新知识
- 新决策 → `wiki/DECISIONS.md`，新概念 → `wiki/CONCEPTS.md`，新模式 → `wiki/PATTERNS.md`
- 写入会话摘要到 `wiki/sessions/YYYY-MM-DD.md`
- 更新 `wiki/INDEX.md` 和 `wiki/LOG.md`

## 验证与检查清单

### 代码提交前检查
- [ ] 是否遵循了"先想再写"原则，明确了所有假设？
- [ ] 代码是否足够简洁，没有过度工程化？
- [ ] 变更是否精准，只修改了必要的内容？
- [ ] 任务是否有明确的成功标准和验证步骤？
- [ ] 是否匹配了项目的代码风格和约定？
- [ ] 是否通过了所有相关测试？
- [ ] 是否更新了必要的文档？

### 工作流变更检查
- [ ] 是否考虑了Athena工作流的稳定性影响？
- [ ] MAREF工作流变更是否经过河图策略验证？
- [ ] 基因管理系统变更是否保持单一职责原则？
- [ ] 是否与现有监控告警系统集成？
- [ ] 是否有回滚计划？

### Wiki 知识蒸馏检查（会话结束时）
- [ ] 本次会话是否产生了新的概念或知识？
- [ ] 是否做了架构决策？→ 更新 `wiki/DECISIONS.md`
- [ ] 是否识别到重复模式？→ 更新 `wiki/PATTERNS.md`
- [ ] 是否需要更新 `wiki/ARCHITECTURE.md`？
- [ ] 是否写了会话摘要到 `wiki/sessions/`？

## 故障排除与最佳实践

### 常见问题解决
1. **代码复杂度过高**: 回到原则2，重新审视是否可以简化
2. **变更范围过大**: 应用原则3，缩小变更范围，分步实施
3. **目标不明确**: 应用原则4，重新定义可验证的成功标准
4. **假设不清晰**: 应用原则1，明确陈述所有假设和不确定性

### 性能优化建议
1. **增量变更**: 小步快跑，每次只做一个明确的变更
2. **持续验证**: 每个小步骤都要有验证机制
3. **及时反馈**: 遇到问题及时寻求反馈，不要隐藏困惑
4. **文档更新**: 代码变更的同时更新相关文档
