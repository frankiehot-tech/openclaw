## 2. 首轮优先级分层

### 2.1 P0 优先级（最高优先级）

**定义**: 门禁任务和高风险动作验证，必须首轮通过才能建立信心基线。

**包含任务**:
1. **ORG-001** (Gate 6A): 桌面状态确认与恢复
   - **重要性**: 所有整理类动作的前提条件，门禁机制核心
   - **通过标准**: 必须通过ADB+视觉双重验证，5个专用字段全部正确
   - **最低样本要求**: 至少3个成功样本（覆盖不同起始状态）

2. **ORG-002** (Gate 6B): 单图标安全整理
   - **重要性**: 验证高风险动作（拖拽）在受控条件下的执行能力
   - **通过标准**: 严格遵循Gate 6A前置，仅执行允许的三种保守动作之一
   - **最低样本要求**: 至少2个成功样本（不同保守动作类型）

3. **ORG-003**: 图标移动到屏幕底部区域
   - **重要性**: 验证目标位置具体化执行能力，测试视觉确认目标区域
   - **通过标准**: 图标准确移动到屏幕底部区域，位置可视觉验证
   - **最低样本要求**: 至少2个成功样本（不同图标和起始位置）

**P0层执行纪律**:
- 必须在P1/P2任务之前执行（ORG-001必须在所有任务之前执行）
- 连续失败3次立即停止首轮回归，进行根本原因分析
- 每个P0任务必须达到最低样本要求才能继续后续P0任务
- P0全部通过前，不得把重点转向P2任务

### 2.2 P1 优先级（中等优先级）

**定义**: 基础功能验证任务，确保记录体系和基本能力可用。

**包含任务**:
- **NAV-001**: 基础返回导航（验证返回键）
- **NAV-002**: 主页键导航（验证主页键+Gate 6A记录）
- **NAV-004**: 通知栏操作（验证系统状态交互+Gate 6A记录）
- **FIND-001**: Wi-Fi设置查找（基础查找能力）
- **FIND-002**: 蓝牙设置查找（相似UI区分）

**P1层执行纪律**:
- 在P0任务通过后按顺序执行
- 允许单任务跳过（仅当明确技术阻塞时），但需记录跳过原因
- 连续失败2次可暂时跳过，但必须在P0全部通过后重新尝试
- 每个任务至少需要1个成功样本才能视为"已验证"

### 2.3 P2 优先级（最低优先级）

**定义**: 扩展功能验证任务，覆盖更复杂场景。

**包含任务**:
- **NAV-005**: 设置内层级导航
- **NAV-003**: 多任务切换
- **FIND-003**: 声音设置查找
- **FIND-004**: 显示设置查找
- **FIND-005**: 应用管理查找

**P2层执行纪律**:
- 仅在P0全部通过且P1基本通过后执行
- 允许在首轮跳过（当时间/资源有限时）
- 跳过决策必须基于明确的技术或资源约束
- 跳过任务必须在回归报告中明确标注，并计划在后续轮次补测

### 2.4 优先级分层依据

1. **门禁优先**: Gate 6A是所有高风险动作的门禁，必须最先验证
2. **依赖优先**: 依赖其他任务的任务排在后面（如Gate 6B依赖Gate 6A）
3. **复杂度递增**: 从简单导航到复杂查找，逐步增加复杂度
4. **风险递增**: 从无风险导航到低风险查找，再到高风险整理
5. **记录体系验证优先**: 优先执行需要验证记录模板和taxonomy的任务

## 3. 首轮执行停止与切换规则

### 3.1 停止规则（硬停止条件）

**立即停止条件**（满足任一条件即停止首轮回归）:
1. **P0门禁连续失败**: ORG-001 (Gate 6A) 连续失败3次
2. **高风险动作违规**: 未通过Gate 6A前置验证即尝试拖拽动作
3. **安全边界突破**: 执行明确禁止的高风险动作（新建文件夹、连续拖拽等）
4. **系统状态异常**: ADB与视觉验证持续冲突（连续3次状态不匹配）
5. **核心服务不可用**: 视觉服务(Qwen)或ADB服务连续失败5次

**暂停条件**（暂停当前任务，尝试恢复后继续）:
1. **单任务连续失败**: 同一任务连续失败2次
2. **临时服务异常**: 视觉服务或ADB服务单次失败
3. **状态确认超时**: 单个状态确认步骤超时（>30秒）

### 3.2 切换规则（任务间切换逻辑）

**正常切换条件**:
1. **任务成功完成**: 任务达到预期终止状态，记录完整
2. **达到最低样本要求**: P0任务达到最低成功样本数（ORG-001: 3个，ORG-002/003: 2个）
3. **记录验证通过**: 执行记录通过格式和完整性检查

**异常切换条件**:
1. **任务跳过**: 明确技术阻塞时，记录跳过原因后切换到下一任务
2. **降级切换**: P2任务因资源限制跳过，切换到下一个可用任务
3. **优先级重排**: 当发现严重问题时，重新评估后续任务优先级

**切换检查点**:
1. **P0→P1切换检查**:
   - [ ] ORG-001 至少3个成功样本
   - [ ] 5个专用字段验证机制确认可靠
   - [ ] ADB+视觉双重验证无持续冲突

2. **P1→P2切换检查**:
   - [ ] 所有P1任务至少1个成功样本
   - [ ] 记录模板使用正确，taxonomy分类合规
   - [ ] 无持续的系统级问题

### 3.3 最低推进条件

**整体推进条件**（必须满足所有条件才能继续下一优先级层）:

1. **P0层推进到P1层的条件**:
   - [ ] ORG-001: 3个成功样本，5个专用字段100%正确
   - [ ] ORG-002: 2个成功样本，拖拽权限记录完整
   - [ ] ORG-003: 2个成功样本，目标区域视觉确认可靠
   - [ ] 高风险动作权限状态记录机制验证有效
   - [ ] 无安全边界突破记录

2. **P1层推进到P2层的条件**:
   - [ ] 所有P1任务至少1个成功样本
   - [ ] 记录模板必填字段100%完整
   - [ ] 失败taxonomy分类100%合规
   - [ ] Gate 6A前置记录机制验证有效
   - [ ] 无持续的技术阻塞问题

3. **首轮回归完成条件**:
   - [ ] 所有P0任务达到最低样本要求
   - [ ] 所有P1任务至少1个成功样本
   - [ ] 记录体系验证通过（字段完整、taxonomy合规、证据链闭合）
   - [ ] 安全机制验证通过（无边界突破、权限记录完整）

### 3.4 异常处理流程

**遇到失败时的标准处理流程**:
1. **记录失败**: 严格按taxonomy标准记录失败（`[主类代码] 具体子因`）
2. **分析原因**: 基于失败类型确定根本原因（感知失败、动作失败、状态不匹配等）
3. **决定行动**:
   - **可恢复失败**: 重试当前任务（最多重试2次）
   - **技术阻塞**: 记录跳过原因，切换到下一任务
   - **系统级问题**: 暂停执行，进行根本原因分析
4. **更新状态**: 在回归跟踪表中更新任务状态

**重试策略**:
- **首次失败**: 立即重试（相同起始条件）
- **第二次失败**: 分析原因，调整参数后重试
- **第三次失败**: 标记为"阻塞"，决定跳过或停止

### 3.5 冻结执行纪律

1. **顺序冻结**: 严格按照本文件定义的执行顺序执行，禁止自行调整
2. **优先级冻结**: P0/P1/P2分层固定，禁止在首轮动态调整优先级
3. **停止规则冻结**: 停止条件严格执行，禁止在触发停止条件后继续执行
4. **记录冻结**: 所有执行记录实时生成，禁止任何后补修改
5. **决策透明**: 所有跳过、暂停、停止决策必须详细记录原因
