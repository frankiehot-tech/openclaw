## 1. 修正清单记录格式

### 1.1 修正项标准字段定义

每条修正项必须包含以下13个标准字段，确保信息完整性和可追溯性：

| 字段序号 | 字段名称 | 字段类型 | 必填性 | 格式说明 | 示例 |
|---------|---------|---------|--------|----------|------|
| 1 | **修正项编号** | 字符串 | 必填 | `FIX-[任务类别]-[三位序号]` | `FIX-ORG-001`, `FIX-NAV-003` |
| 2 | **触发日期** | 日期 | 必填 | `YYYY-MM-DD` | `2026-04-03` |
| 3 | **关联任务编号** | 字符串 | 必填 | `[类别]-[序号]` | `ORG-001`, `NAV-002` |
| 4 | **关联任务名称** | 字符串 | 必填 | 任务完整名称 | `桌面状态确认与恢复` |
| 5 | **问题类型** | 枚举 | 必填 | 见1.2节问题类型枚举 | `task_definition_issue` |
| 6 | **问题描述** | 文本 | 必填 | 详细问题描述，包含具体场景 | `任务目标描述存在歧义，可理解为两种不同操作` |
| 7 | **触发样本编号** | 字符串 | 必填 | 触发问题的样本编号 | `S001`, `S005`, `S012` |
| 8 | **问题影响范围** | 枚举 | 必填 | `single_sample`/`task_level`/`category_level`/`system_level` | `task_level` |
| 9 | **建议修正内容** | 文本 | 必填 | 具体的修正方案，包含修改前后的对比 | `将"整理图标"改为"将'设置'图标移动到屏幕底部区域"` |
| 10 | **是否影响冻结基线** | 布尔 | 必填 | `true`/`false` | `false` |
| 11 | **是否需要下周处理** | 布尔 | 必填 | `true`/`false` | `true` |
| 12 | **当前状态** | 枚举 | 必填 | `pending`/`under_review`/`approved`/`rejected`/`implemented` | `pending` |
| 13 | **备注** | 文本 | 可选 | 额外说明、评审意见、处理建议 | `需与安全团队确认修改后的权限要求` |

### 1.2 问题类型固定枚举

问题类型必须使用以下固定枚举值，禁止自定义或使用其他描述：

| 问题类型代码 | 问题类型名称 | 问题类型定义 | 典型场景 |
|-------------|------------|-------------|----------|
| `task_definition_issue` | 任务定义问题 | 任务目标、输入、输出、成功标准定义不明确或不完整 | 目标描述歧义，成功标准模糊 |
| `template_issue` | 模板问题 | 执行记录模板字段缺失、冗余或格式不正确 | 缺少必填字段，字段命名不一致 |
| `taxonomy_issue` | 分类问题 | 失败分类标准不适用、覆盖不全或存在重叠 | 新失败场景无对应分类 |
| `permission_rule_issue` | 权限规则问题 | 高风险动作权限要求不合理或执行困难 | 拖拽权限记录要求过于严格 |
| `gate_rule_issue` | 门控规则问题 | Gate验证条件不合理或验证逻辑存在漏洞 | Gate 6A验证条件不完整 |
| `sample_management_issue` | 样本管理问题 | 样本编号、命名、存储、统计规则存在问题 | 样本编号重复，存储目录混乱 |
| `execution_sequence_issue` | 执行顺序问题 | 任务执行顺序依赖关系不合理 | 前置任务未完成即执行依赖任务 |
| `resource_constraint_issue` | 资源约束问题 | 任务对设备、网络、计算资源要求不合理 | 任务需要特定设备状态但未定义 |
| `safety_boundary_issue` | 安全边界问题 | 安全边界定义不明确或保护措施不足 | 高风险动作缺少必要的前置检查 |
| `visual_evidence_issue` | 视觉证据问题 | 截图要求、质量、数量标准不合理 | 关键步骤缺少截图要求 |

### 1.3 修正清单标准格式示例

**修正项示例**：

```
## 修正项: FIX-ORG-002

| 字段 | 值 |
|------|-----|
| **修正项编号** | FIX-ORG-002 |
| **触发日期** | 2026-04-03 |
| **关联任务编号** | ORG-002 |
| **关联任务名称** | 单图标安全整理 |
| **问题类型** | `task_definition_issue` |
| **问题描述** | 任务目标"整理图标"存在歧义，未指定具体整理哪个图标。执行时可能选择任意图标，导致样本不一致。 |
| **触发样本编号** | S003, S007, S009 |
| **问题影响范围** | `task_level` (影响ORG-002所有样本) |
| **建议修正内容** | 将目标描述修改为："将'设置'图标从当前位置移动到桌面第一页的空白区域"。明确指定图标名称和移动目标。 |
| **是否影响冻结基线** | `false` |
| **是否需要下周处理** | `true` |
| **当前状态** | `pending` |
| **备注** | 需确认'设置'图标在所有测试设备上的存在性和位置一致性。 |
```

## 2. 修正清单触发规则

### 2.1 必须触发修正清单的10类情形

以下10类情形必须触发修正清单记录，无例外情况：

| 情形编号 | 触发情形 | 问题类型映射 | 触发条件 | 处理优先级 |
|---------|----------|-------------|----------|------------|
| **TR-01** | **任务目标歧义** | `task_definition_issue` | 同一任务目标存在2种或以上合理理解 | P0 (必须本周处理) |
| **TR-02** | **执行路径缺失关键步骤** | `task_definition_issue` | 任务缺少必要的前置检查或状态验证 | P0 |
| **TR-03** | **成功标准与实际情况冲突** | `task_definition_issue` | 定义的成功标准无法通过正常执行达成 | P0 |
| **TR-04** | **模板字段缺失影响统计** | `template_issue` | 必填字段缺失导致样本无法纳入统计 | P1 (建议本周处理) |
| **TR-05** | **失败场景无对应分类** | `taxonomy_issue` | 新失败原因不在现有taxonomy覆盖范围内 | P1 |
| **TR-06** | **权限要求不合理影响执行** | `permission_rule_issue` | 权限记录要求过于复杂或无法实现 | P1 |
| **TR-07** | **Gate验证逻辑存在漏洞** | `gate_rule_issue` | Gate可能通过但实际状态不满足安全要求 | P0 |
| **TR-08** | **样本编号分配冲突** | `sample_management_issue` | 样本编号重复或分配规则冲突 | P1 |
| **TR-09** | **执行顺序依赖错误** | `execution_sequence_issue` | 任务执行顺序导致状态冲突 | P1 |
| **TR-10** | **安全边界保护不足** | `safety_boundary_issue` | 高风险动作缺少必要的前置检查 | P0 |

### 2.2 触发条件详细说明

#### TR-01: 任务目标歧义
**检测方法**: 双人独立解读任务目标，比较解读结果
**触发阈值**: 2名评审员对目标理解不一致
**示例**: "整理桌面图标" - 可能理解为整理所有图标或整理单个图标

#### TR-02: 执行路径缺失关键步骤
**检测方法**: 执行路径回溯分析，检查状态转移完整性
**触发阈值**: 缺少从起始状态到目标状态的必要中间步骤
**示例**: 拖拽任务缺少桌面状态确认步骤

#### TR-03: 成功标准与实际情况冲突
**检测方法**: 成功标准与设备实际状态对比
**触发阈值**: 定义的成功状态在实际设备上无法达成
**示例**: 要求特定图标在固定位置，但不同设备图标位置不同

#### TR-04: 模板字段缺失影响统计
**检测方法**: 模板字段与统计字段映射检查
**触发阈值**: 统计必需字段在模板中缺失或无法获取
**示例**: 成功率统计需要"执行时长"字段但模板中未定义

#### TR-05: 失败场景无对应分类
**检测方法**: 失败原因与现有taxonomy匹配检查
**触发阈值**: 失败原因无法归入任何现有主类+子因组合
**示例**: 新设备兼容性问题导致的失败

#### TR-06: 权限要求不合理影响执行
**检测方法**: 权限记录实际操作可行性评估
**触发阈值**: 权限记录步骤增加>30%执行时间或无法自动化实现
**示例**: 每次拖拽前都需要手动确认权限

#### TR-07: Gate验证逻辑存在漏洞
**检测方法**: Gate验证逻辑与安全要求对比分析
**触发阈值**: Gate可能返回true但实际存在安全风险
**示例**: Gate 6A可能通过但桌面实际存在弹窗

#### TR-08: 样本编号分配冲突
**检测方法**: 样本编号唯一性检查
**触发阈值**: 发现重复样本编号或编号分配规则冲突
**示例**: 并行执行导致样本编号重复

#### TR-09: 执行顺序依赖错误
**检测方法**: 任务依赖关系图分析
**触发阈值**: 前置任务未完成但后续任务已开始执行
**示例**: ORG-001未完成即执行ORG-002

#### TR-10: 安全边界保护不足
**检测方法**: 安全边界覆盖度分析
**触发阈值**: 高风险动作缺少至少一项必要的前置检查
**示例**: 拖拽动作缺少桌面锁定状态检查

### 2.3 不进入修正清单的情形

以下情形**不触发**修正清单记录，按其他流程处理：

| 情形编号 | 情形描述 | 处理方式 | 理由 |
|---------|----------|----------|------|
| **NT-01** | **单纯执行失败** | 按失败taxonomy分类 | 属于技术能力问题，非定义问题 |
| **NT-02** | **执行过程违规** | 按违规样本流程处理 | 违反既定规则，非定义缺陷 |
| **NT-03** | **设备环境问题** | 记录为环境异常样本 | 外部因素导致，非定义问题 |
| **NT-04** | **模型能力限制** | 记录为技术限制 | 当前技术能力边界，非定义错误 |
| **NT-05** | **临时性网络故障** | 记录为执行异常 | 临时性因素，非系统性定义问题 |
| **NT-06** | **人为操作失误** | 记录为操作失误 | 执行者问题，非定义缺陷 |
| **NT-07** | **已记录的已知问题** | 引用已有修正项 | 避免重复记录 |
| **NT-08** | **建议性优化（非缺陷）** | 记录为优化建议 | 非定义问题，属于改进建议 |

### 2.4 触发流程

**标准触发流程**：

```
步骤1: 问题识别
├─ 执行过程中发现定义问题
├─ 或评审过程中识别问题
└─ 初步判断是否符合TR-01至TR-10情形

步骤2: 问题确认
├─ 收集问题证据（截图、日志、样本记录）
├─ 确认问题可复现性（至少2个样本）
└─ 评估问题影响范围

步骤3: 修正项创建
├─ 生成修正项编号（FIX-[类别]-[序号]）
├─ 填写13个标准字段
└─ 设置初始状态为`pending`

步骤4: 优先级评估
├─ 根据情形编号确定处理优先级（P0/P1）
├─ 评估是否影响冻结基线
└─ 决定是否需要下周处理

步骤5: 记录归档
├─ 将修正项记录到对应类别的修正清单文件
├─ 更新修正清单索引
└─ 通知相关评审人员
```

### 2.5 优先级处理规则

**优先级定义**：
- **P0 (必须本周处理)**: 影响任务可执行性、安全性或数据统计有效性
- **P1 (建议本周处理)**: 影响执行效率、记录完整性或长期质量

**处理时限**：
- **P0问题**: 发现后24小时内必须开始处理
- **P1问题**: 发现后72小时内开始处理

**升级机制**：
- 超过时限未处理的P0问题自动升级为项目阻塞项
- 超过时限未处理的P1问题自动升级为P0问题

## 3. 修正清单与冻结基线隔离规则

### 3.1 核心隔离原则

**不可违背的核心规则**：首轮执行中**不得直接覆盖**第1周冻结基线文件

**隔离层级**：
1. **物理文件隔离**: 修正清单存储在Week2目录，基线文件存储在Week1目录
2. **版本标识隔离**: 所有修正项明确标识`Week2`，与`Week1`基线区分
3. **引用关系隔离**: 修正项引用基线文件时使用只读引用，禁止修改

### 3.2 基线文件保护清单

**受保护的Week1冻结基线文件**：

| 基线文件 | 保护级别 | 允许的操作 | 禁止的操作 |
|---------|---------|-----------|-----------|
| `Athena_v1.1_Week1_Baseline_Package.md` | 最高级 | 只读引用 | 任何修改 |
| `Athena_v1.1_回归测试集（首版）.md` | 最高级 | 只读引用，创建修正项 | 直接修改任务定义 |
| `Athena_v1.1_任务级执行记录映射表.md` | 高级 | 只读引用，记录模板问题 | 直接修改字段映射 |
| `Athena统一测试记录模板.md` | 高级 | 只读引用，创建模板修正 | 直接修改模板结构 |
| `Athena失败taxonomy冻结版.md` | 高级 | 只读引用，创建分类修正 | 直接修改分类标准 |
| `Athena高风险动作清单与权限口径.md` | 高级 | 只读引用，创建权限修正 | 直接修改权限规则 |
| `gate_6_unified_execution_spec.md` | 最高级 | 只读引用，创建门控修正 | 直接修改门控逻辑 |
| `Athena_v1.1_首轮回归执行顺序与优先级.md` | 高级 | 只读引用，创建顺序修正 | 直接修改执行顺序 |

### 3.3 修正项与基线关系处理规则

#### 规则1: 引用而非修改
**操作方式**: 在修正项中引用基线文件的特定章节或定义
**示例**: "参考《Athena_v1.1_回归测试集（首版）.md》第15页ORG-002定义"

#### 规则2: 版本化修正建议
**操作方式**: 建议的修正内容必须明确标注为"v1.2建议"或"第3周建议"
**示例**: "建议v1.2版本将任务目标修改为：..."

#### 规则3: 影响基线评估
**操作方式**: 每个修正项必须评估是否影响冻结基线
**判断标准**: 
- `true`: 修正内容需要修改基线文件定义
- `false`: 修正内容不涉及基线文件修改

#### 规则4: 处理时机控制
**操作方式**: 通过"是否需要下周处理"字段控制修正实施时机
**处理逻辑**:
- `true`: 问题需要在下周（第3周）处理，本周仅记录
- `false`: 问题可在本周（第2周）立即处理

### 3.4 边界定义与交互规则

#### 3.4.1 修正清单 vs 失败taxonomy

| 维度 | 修正清单 | 失败taxonomy |
|------|---------|-------------|
| **目的** | 记录任务定义问题 | 记录执行技术失败 |
| **触发时机** | 定义发现缺陷时 | 任务执行失败时 |
| **内容** | 定义修正建议 | 失败原因分类 |
| **处理** | 定义优化迭代 | 技术问题分析 |
| **边界规则** | 当失败原因是定义问题时，先记录修正项，再按技术失败分类 |

#### 3.4.2 修正清单 vs 违规样本记录

| 维度 | 修正清单 | 违规样本记录 |
|------|---------|-------------|
| **目的** | 记录定义缺陷 | 记录规则违反 |
| **触发条件** | 定义问题 | 执行违规 |
| **责任归属** | 定义制定者 | 执行者 |
| **处理方式** | 定义修正 | 违规处置 |
| **边界规则** | 当违规原因是定义模糊时，先记录修正项，再记录违规 |

#### 3.4.3 修正清单 vs 执行记录

| 维度 | 修正清单 | 执行记录 |
|------|---------|----------|
| **目的** | 记录定义问题 | 记录执行过程 |
| **内容类型** | 元数据问题 | 具体执行数据 |
| **使用场景** | 定义优化 | 执行审计 |
| **边界规则** | 执行记录中发现问题时，创建独立的修正项记录 |

#### 3.4.4 修正清单 vs 回归汇总表

| 维度 | 修正清单 | 回归汇总表 |
|------|---------|------------|
| **目的** | 记录定义问题 | 汇总执行结果 |
| **内容** | 问题清单 | 统计指标 |
| **更新频率** | 发现问题时 | 执行完成时 |
| **边界规则** | 修正项中的问题影响统计时，在汇总表中添加备注说明 |

### 3.5 修正实施流程

#### 第2周（首轮回归）修正处理：

```
流程: 仅记录，不实施
├─ 发现定义问题 → 创建修正项
├─ 状态标记为`pending`或`under_review`
├─ 评估影响和优先级
├─ 记录建议修正内容
└─ 等待第3周统一处理
```

#### 第3周（稳定性统计）修正实施：

```
流程: 评审与实施
├─ 评审第2周收集的所有修正项
├─ 批准通过的修正项（状态改为`approved`）
├─ 实施不影响基线的修正（立即应用）
├─ 标记影响基线的修正（计划v1.2实施）
└─ 更新修正项状态为`implemented`或`rejected`
```

#### 基线文件修改流程（第4周及以后）：

```
流程: 基线版本迭代
├─ 收集所有`approved`且影响基线的修正项
├─ 制定基线更新计划
├─ 创建v1.2基线草案
├─ 评审与批准v1.2基线
└─ 正式发布新基线，原基线归档
```

### 3.6 审计与合规检查

#### 每日检查项：
- [ ] 是否有修正项直接引用但修改了基线文件内容
- [ ] 所有修正项是否正确评估"是否影响冻结基线"
- [ ] 修正项存储位置是否符合目录规范
- [ ] P0问题是否在时限内开始处理

#### 每周检查项：
- [ ] 修正项数量与问题类型分布
- [ ] 影响基线的修正项占比
- [ ] 处理及时性统计
- [ ] 修正项与执行失败的关联分析

#### 合规违规处理：
1. **发现直接修改基线文件**: 立即回滚，记录严重违规
2. **修正项信息不完整**: 暂停相关任务执行，补充完整
3. **未按时处理P0问题**: 升级为项目风险，负责人问责
4. **修正项存储不规范**: 重新组织，记录规范违反
