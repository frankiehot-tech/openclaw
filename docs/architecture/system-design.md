# Athena队列系统架构文档

## 概述
Athena队列系统是一个智能任务编排和工作流管理系统，负责管理、执行和监控AI计划任务。系统采用契约优先的设计原则，确保组件的可靠交互和状态一致性。

## 系统架构设计

### 核心组件

#### 1. 任务队列管理器 (Queue Manager)
- **功能**: 管理任务队列状态，包括pending、running、completed、failed、manual_hold等状态
- **关键文件**: `/Volumes/1TB-M2/openclaw/.openclaw/plan_queue/openhuman_aiplan_build_priority_20260328.json`
- **状态管理**: 
  - `queue_status`: running, dependency_blocked, empty, failed, manual_hold
  - `pause_reason`: 记录暂停原因
  - `current_item_id`: 当前正在执行的任务ID

#### 2. 队列运行器 (Queue Runner)
- **功能**: 执行队列中的任务，管理任务生命周期
- **关键脚本**: `/Volumes/1TB-M2/openclaw/scripts/athena_ai_plan_runner.py`
- **核心逻辑**:
  - `materialize_route_items()`: 从manifest生成可执行任务
  - `compute_route_counts_and_status()`: 计算队列状态和计数
  - `spawn_build_worker()`: 启动构建工作进程
  - `detect_and_cleanup_stale_runs()`: 检测和清理僵尸进程

#### 3. 监控系统 (Monitoring System)
- **监控仪表板**: `queue_monitor_dashboard.py` (端口5002)
  - 实时显示队列状态和告警信息
  - API端点: `/api/status`, `/api/alerts`, `/api/metrics`
- **监控守护进程**: `two_minute_queue_monitor.py`
  - 每2分钟检查队列状态
  - 生成告警和指标
- **队列监控器**: `queue_monitor.py --daemon`
  - 持续监控队列状态变化

#### 4. 数据存储
- **Manifest文件**: `openhuman_aiplan_priority_execution_20260414.json`
  - 211个任务条目（160个唯一ID，51个重复）
  - 包含任务元数据、依赖关系、执行配置
- **队列状态文件**: `openhuman_aiplan_build_priority_20260328.json`
  - 存储当前队列状态和任务状态
  - 与manifest同步状态信息

### 架构演进：从传统架构到新架构

#### 传统架构问题
1. **状态管理分散**: 状态信息分散在Web界面、队列文件、manifest中
2. **错误处理脆弱**: 复杂的错误恢复逻辑容易出错
3. **资源管理不足**: 并行构建门控实现不完整
4. **监控与执行耦合**: 监控系统与执行系统紧密耦合
5. **数据一致性挑战**: Manifest、队列状态、任务状态之间缺乏强一致性保证

#### 新架构改进
1. **契约优先设计**: 明确定义组件间的交互契约
2. **单一事实源**: 统一状态管理，消除状态分散
3. **故障隔离**: 组件间故障不传播，系统具备优雅降级能力
4. **可观测性**: 系统状态完全透明，支持深度调试和监控
5. **渐进式演进**: 重构分阶段进行，保持系统可用性

## 关键设计决策

### 1. 任务身份规范化
- **问题**: 任务ID以`-`开头被`argparse`误识别为flag参数
- **解决方案**: 建立`TaskIdentityContract`契约
  - 生成不以`-`开头的规范化任务ID
  - 提供原始ID到规范化ID的转换
  - 确保与argparse参数解析兼容

### 2. 进程生命周期管理
- **问题**: `spawn_build_worker`函数先标记状态为running再启动进程
- **解决方案**: 建立`ProcessLifecycleContract`契约
  - 先启动进程，成功后再更新状态
  - 减少心跳检测延迟（从5分钟到30秒）
  - 增强僵尸进程检测

### 3. 状态同步机制
- **问题**: Web界面、队列文件、manifest状态不一致
- **解决方案**: 建立`StateSyncContract`契约
  - 原子性状态更新，确保一致性
  - 乐观锁机制防止并发冲突
  - 自动同步到manifest和Web界面

### 4. 智能工作流编排
- **问题**: Claude Code CLI与OpenCode执行器混淆使用
- **解决方案**: 建立`SmartOrchestrator`智能路由
  - 基于entry_stage的明确路由规则
  - 考虑系统负载和预算状态的自适应策略
  - 执行器类型明确区分（Claude Code CLI, OpenCode Build等）

## 数据流和状态转换

### 任务生命周期
```
创建任务 → 依赖检查 → 等待执行 → 执行中 → 完成/失败
```

### 队列状态转换
```
empty → running (有pending任务且无阻塞)
running → dependency_blocked (发现依赖阻塞)
dependency_blocked → running (依赖解决)
running → empty (所有任务完成)
empty → manual_hold (手动暂停)
```

### 状态同步流程
1. **队列运行器**更新队列文件状态
2. **状态同步契约**原子性更新manifest和Web缓存
3. **监控系统**检测状态变化并生成告警
4. **Web界面**实时显示最新状态

## 依赖管理

### 依赖类型
1. **硬依赖**: 必须等待前置任务完成才能开始
2. **软依赖**: 可以并行执行，但需要共享资源
3. **幽灵依赖**: 指向不存在任务的依赖关系（需要清理）

### 依赖解析算法
1. **广度优先遍历**依赖图
2. **检测循环依赖**和幽灵依赖
3. **并行执行**无依赖关系的任务
4. **依赖满足检查**在任务启动前验证

## 错误处理和恢复

### 错误分类
1. **进程错误**: 工作进程启动失败、异常退出
2. **依赖错误**: 缺失依赖、循环依赖、幽灵依赖
3. **资源错误**: 内存不足、磁盘空间不足、CPU过载
4. **状态错误**: 状态不一致、数据损坏

### 恢复策略
1. **自动重试**: 对临时性错误自动重试（最多3次）
2. **依赖修复**: 自动检测和修复幽灵依赖
3. **状态修复**: 使用修复脚本恢复状态一致性
4. **优雅降级**: 在资源不足时降低并发度
