# 文档写作指南

## 概述
本文档提供OpenClaw项目文档写作的最佳实践和标准规范。遵循这些指南可以确保文档的一致性、可读性和实用性。

## 核心原则

### 1. 用户中心原则
- **目标读者明确**: 每篇文档都应明确目标读者（开发者、用户、维护者）
- **解决问题导向**: 文档应解决特定问题或完成特定任务
- **实用价值优先**: 优先提供可直接操作的信息，理论解释为辅

### 2. 一致性原则
- **术语统一**: 使用项目标准术语表中的定义
- **格式规范**: 遵循Markdown格式规范和模板结构
- **风格统一**: 保持语言风格和技术深度的一致性

### 3. 可维护性原则
- **模块化结构**: 文档应模块化，便于单独更新和维护
- **清晰所有权**: 明确文档维护者和更新频率
- **版本控制**: 使用语义化版本控制跟踪文档变更

## 文档结构标准

### 标准文档模板
所有文档应包含以下基本部分：

```markdown
# 文档标题

## 概述
- 文档目的和范围
- 目标读者
- 前置知识要求

## 主要内容
- 按逻辑顺序组织的核心内容
- 使用适当的标题层级（##, ###, ####）

## 实践指南
- 具体操作步骤
- 代码示例和配置
- 最佳实践建议

## 故障排除
- 常见问题及解决方案
- 错误信息解释
- 调试方法

## 相关文档
- 指向相关文档的链接
- 进阶学习资源

## 版本历史
- 变更记录和维护者信息
```

### 标题层级规范
- **# 一级标题**: 文档标题
- **## 二级标题**: 主要章节
- **### 三级标题**: 子章节
- **#### 四级标题**: 小节（尽量少用）

## 内容写作技巧

### 技术文档写作
1. **代码示例**
   - 提供完整可运行的代码片段
   - 包含必要的导入和上下文
   - 添加注释解释关键部分

2. **配置说明**
   - 提供默认配置和示例配置
   - 解释每个配置项的作用
   - 说明配置变更的影响

3. **操作步骤**
   - 使用有序列表（1., 2., 3.）
   - 每一步都有明确的预期结果
   - 包含验证步骤确认操作成功

### 概念解释文档
1. **抽象概念具体化**
   - 使用类比帮助理解
   - 提供实际应用场景
   - 对比相关概念的区别

2. **架构设计文档**
   - 包含系统架构图
   - 解释组件职责和交互
   - 说明设计决策和权衡

## Markdown高级技巧

### 表格使用
```markdown
| 特性 | 描述 | 示例 |
|------|------|------|
| 功能A | 详细描述 | `code example` |
| 功能B | 详细描述 | [链接](https://example.com) |
```

### 代码块语法
````markdown
```python
# Python代码示例
def example():
    return "Hello"
```

```bash
# Shell命令
echo "Hello"
```

```yaml
# 配置文件
key: value
```
````

### 警告和提示框
```markdown
> **注意**: 重要提醒信息

> **警告**: 潜在风险或破坏性操作

> **提示**: 有用的小技巧或优化建议
```

</details>

## 视觉元素指南

### 图片使用
1. **格式要求**: 优先使用PNG格式，支持透明背景
2. **尺寸优化**: 图片宽度不超过1200px，大小不超过500KB
3. **替代文本**: 所有图片必须添加描述性替代文本
4. **存储位置**: 图片存储在`docs/assets/images/`目录

```markdown
![架构图描述](../assets/images/architecture-diagram.png)
```

### 图表和流程图
1. **Mermaid图表**（如支持）:
   ```mermaid
   graph TD
     A[开始] --> B[处理]
     B --> C[结束]
   ```

2. **PlantUML**（如支持）:
   ```plantuml
   @startuml
   Alice -> Bob: 请求
   Bob --> Alice: 响应
   @enduml
   ```

## 文档质量检查清单

### 提交前检查
- [ ] **准确性**: 技术内容准确无误
- [ ] **完整性**: 覆盖主题所有重要方面
- [ ] **一致性**: 术语和格式符合项目标准
- [ ] **可读性**: 语言清晰，结构合理
- [ ] **实用性**: 提供实际可操作的指导
- [ ] **链接有效性**: 所有链接指向有效目标
- [ ] **语法正确**: 无拼写和语法错误
- [ ] **代码可运行**: 代码示例可正确执行
- [ ] **图片显示**: 图片正确加载和显示
- [ ] **元数据完整**: 包含版本、维护者、更新日期

## 协作流程

### 文档评审
1. **技术准确性评审**: 由领域专家验证技术内容
2. **用户体验评审**: 由用户代表验证可理解性
3. **格式规范评审**: 由文档维护者验证格式合规性

### 版本管理
1. **语义化版本**: 使用MAJOR.MINOR.PATCH版本号
2. **变更日志**: 记录所有重大变更和修复
3. **兼容性**: 确保新版本不破坏现有链接和引用

## 常见问题解答

### Q1: 如何平衡技术深度和可读性？
**A**: 采用分层方法：摘要层提供概述，细节层提供深入信息，附录层提供技术细节。使用折叠内容隐藏高级细节。

### Q2: 文档应该多详细？
**A**: 详细到能让目标读者成功完成任务。提供必要的前置条件、步骤、验证方法和故障排除，但避免无关细节。

### Q3: 如何处理过时的文档？
**A**: 标记为过时并添加指向新文档的链接，或直接更新内容。避免删除文档，除非确定不再需要。

### Q4: 如何鼓励团队贡献文档？
**A**: 提供模板、示例、评审支持和认可机制。将文档贡献纳入绩效考核。

### Q5: 如何管理多语言文档？
**A**: 使用标准目录结构（如`docs/zh-cn/`），保持源语言和翻译版本同步，建立翻译评审流程。
