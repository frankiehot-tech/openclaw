# Athena队列系统契约框架工程化阶段性部署计划

## 概述

基于MAREF（多智能体递归进化框架）原则和深度审计结果，Athena队列系统已完成智能工作流契约框架的核心重构。本计划旨在实施工程化的阶段性部署，确保系统在从旧架构向新架构迁移过程中保持稳定性和可用性。

## 部署背景

### 已完成的重构工作
1. ✅ **深度审计**：识别5个系统性设计缺陷和架构根本问题
2. ✅ **契约框架实现**：完成TaskIdentityContract、ProcessLifecycleContract、DataQualityContract、StateSyncContract等核心契约
3. ✅ **智能工作流引擎**：实现SmartOrchestrator，解决15%执行器混淆率
4. ✅ **沙箱验证**：通过端到端集成测试（6/6通过）和智能编排器集成测试（5/5通过）
5. ✅ **监控升级**：完善队列监控仪表板和告警系统

### 解决的核心问题
1. **任务身份规范化失败**：TaskIdentityContract确保ID不以'-'开头，避免argparse误识别
2. **Manifest数据质量缺陷**：DataQualityContract清理24%重复数据，建立数据质量监控
3. **进程可靠性契约缺失**：ProcessLifecycleContract先启动进程再更新状态，优化心跳检测（5分钟→30秒）
4. **状态管理分散**：StateSyncContract实现原子状态更新，确保Web界面、队列文件、manifest一致性
5. **Lane混合与路由混淆**：SmartOrchestrator明确执行器路由规则，消除执行器混淆

## 部署原则

### MAREF框架指导原则
1. **超稳定性要求**：部署过程必须保证系统在状态空间中收敛到稳定区域
2. **三才六层模型**：部署计划覆盖环境层、平台层、应用层、流程层、管理层、战略层
3. **渐进式演进**：分阶段、分批次的迁移策略，确保系统始终处于可控状态
4. **故障隔离**：新架构组件与旧系统隔离运行，故障不传播
5. **可观测性**：部署过程完全透明，所有状态变更可监控、可追溯

### gstack工程化原则
- **质量门禁检查**：每个部署阶段前必须通过严格的质量检查
- **风险控制矩阵**：识别潜在风险并制定缓解措施
- **回滚能力**：每个阶段都具备快速回滚到前一稳定状态的能力
- **文档驱动**：所有决策、配置、验证结果记录在案

## 部署阶段设计

   - 确认监控系统正常运行

2. **数据备份与快照**
   - 备份所有队列文件（`.openclaw/plan_queue/`, `.openclaw/queue_states/`）
   - 备份Manifest文件（`gene_management_queue_manifest.json`等）
   - 创建系统配置快照

3. **契约框架验证**
   - 运行契约框架集成测试：`python test_contract_framework_integration.py`
   - 运行智能编排器集成测试：`python test_smart_orchestrator_integration.py`
   - 验证所有测试通过（6/6, 5/5）

4. **环境准备**
   - 创建隔离测试环境（沙箱）
   - 配置监控仪表板专门用于部署监控
   - 设置部署专用日志和指标收集

#### 质量门禁
- [ ] 所有队列处于稳定状态（无卡死、无大量失败任务）
- [ ] 数据备份完成且验证可恢复
- [ ] 契约框架测试全部通过
- [ ] 监控系统部署就绪

   - 更新任务生成器使用`TaskIdentity.generate()`生成新ID

2. **并行运行验证**

3. **Manifest数据更新**
   - 使用`TaskIdentityContract`批量更新现有Manifest中的问题ID
   - 验证更新后Manifest与队列文件的一致性

4. **Web界面适配**
   - 更新Web界面显示规范化ID，同时显示原始ID（括号内）

#### 回滚检查点

### 阶段2：进程生命周期管理升级（3天）

#### 范围
重构进程启动、监控、清理逻辑，优化心跳检测

#### 实施步骤
1. **重构`spawn_build_worker`函数**
   - 替换为`ProcessContract.spawn()`方法（先启动进程再更新状态）
   - 更新状态机逻辑，确保进程启动失败时状态回滚
   - 修改`detect_and_cleanup_stale_runs`函数（约第2105行）

2. **心跳检测优化**
   - 将`HEARTBEAT_THRESHOLD_MINUTES`从5分钟减少到30秒
   - 实现秒级进程健康检查，快速检测僵尸进程
   - 优化资源清理策略，减少死进程占用时间

3. **进程监控仪表板**
   - 在现有监控仪表板中添加进程生命周期监控
   - 实时显示进程状态、CPU/内存使用、心跳状态
   - 添加进程异常告警（僵尸进程、资源泄漏等）

4. **测试与验证**
   - 模拟进程启动失败场景，验证状态回滚机制
   - 测试心跳检测的敏感性和准确性
   - 验证资源清理效率提升效果

#### 回滚检查点
- 阶段2.2完成：验证心跳检测优化效果
- 阶段2.4完成：进程异常处理测试通过

- [ ] 进程状态与队列状态一致性100%
- [ ] 无资源泄漏或僵尸进程累积

### 阶段3：数据质量与状态同步（2天）

#### 范围

#### 实施步骤

   - 实现乐观锁机制，防止并发更新冲突

3. **一致性验证工具**
   - 开发状态一致性检查工具，定期验证三个状态源的一致性
   - 在监控仪表板中添加一致性状态显示

4. **迁移现有状态数据**
   - 将现有分散的状态数据迁移到统一状态存储
   - 验证迁移后状态完整性和历史可追溯性

### 阶段4：智能工作流引擎上线（3天）

#### 范围
部署SmartOrchestrator，实现智能路由决策

#### 实施步骤

   - 确保向后兼容性（SmartOrchestrator不可用时使用基础路由）

2. **执行器生态系统规范化**
   - 明确10种执行器类型（Claude Code CLI, OpenCode Build等）的边界
   - 更新执行器适配层，统一接口规范

4. **性能监控与优化**
   - 监控智能路由决策质量（混淆率、决策时间、成本效率）
   - A/B测试对比智能路由与基础路由效果

### 阶段6：生产环境全面切换（1天）

#### 范围
将生产流量完全切换到新架构，完成部署

#### 实施步骤
1. **流量切换准备**
   - 确认所有质量门禁检查通过
   - 准备切换脚本和回滚脚本
   - 通知相关团队和用户

2. **渐进式流量切换**
   - 首先切换10%生产流量到新架构
   - 监控关键指标，确认系统稳定
   - 逐步增加切换比例（30%→50%→80%→100%）

3. **最终验证与监控**
   - 100%流量切换后运行完整验证测试
   - 监控系统24小时，确保无隐藏问题
   - 收集用户反馈，验证功能完整性

4. **部署后优化**
   - 根据生产环境数据优化配置参数
   - 调整监控告警阈值
   - 更新运维文档和故障处理指南

#### 回滚计划
- 任何时候发现问题，可快速回滚到前一阶段
- 保留完整的旧系统备份，支持完全回退
- 回滚决策标准：关键指标恶化超过10%，或出现影响用户的功能问题

1. **系统健康指标**

   - 任务处理延迟（P50、P90、P99）

4. **业务指标**
   - 用户任务完成率
   - 手动拉起成功率
   - 用户满意度（如可收集）
