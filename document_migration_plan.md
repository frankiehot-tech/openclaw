# OpenClaw项目文档迁移计划

## 概述

基于路径配置优化项目（4个优化建议中的第2项），本计划旨在重新组织OpenClaw项目文档，建立标准化的文档管理体系，支持项目的长期维护和知识传承。

## 现状分析

### 当前文档结构问题
1. **文档分散**：文档分散在项目根目录、skills目录、vendor目录等多个位置
2. **缺乏分类**：核心架构文档、审计报告、部署指南、用户文档混合存放
3. **命名不一致**：中文/英文混合命名，日期格式不统一
4. **模板文件未使用**：Claude Code标准模板文件（IDENTITY.md, USER.md等）大部分为空
5. **技术债务**：大量审计报告和临时分析文档缺乏组织

### 关键文档识别
| 文档类型 | 示例文件 | 数量估计 | 重要性 |
|----------|----------|----------|--------|
| **核心架构文档** | COGNITIVE_DNA.md, AGENTS.md | 5-10 | 高 |
| **技术规范** | athena_agent_roles_pressure_analysis.md | 20+ | 中高 |
| **部署指南** | engineering_staged_deployment_plan.md, deployment_guide.md | 5-10 | 高 |
| **审计报告** | aiplan_queue_monitoring_final_analysis_report_20260419.md | 30+ | 中 |
| **用户指南** | USER.md, TOOLS.md（模板） | 5-10 | 中低 |
| **技能文档** | skills/*/SKILL.md | 10+ | 中 |
| **Vendor文档** | vendor/tenacitOS/*.md | 50+ | 低 |

## 目标架构

### 文档分类体系
基于MAREF三才六层模型和gstack文档驱动原则，建立三层文档架构：

```
openclaw/
├── docs/                          # 文档根目录
│   ├── architecture/              # 架构文档（战略层、管理层）
│   │   ├── cognitive-dna.md       # 认知DNA（核心）
│   │   ├── agents-overview.md     # 智能体架构
│   │   ├── system-design.md       # 系统设计文档
│   │   └── marer-framework.md     # MAREF框架应用指南
│   │
│   ├── technical/                 # 技术文档（平台层、应用层）
│   │   ├── specifications/        # 技术规范
│   │   │   ├── athena-spec.md
│   │   │   ├── queue-spec.md
│   │   │   └── api-spec.md
│   │   ├── deployment/           # 部署指南
│   │   │   ├── staged-deployment-plan.md
│   │   │   ├── production-deployment.md
│   │   │   └── rollback-guide.md
│   │   └── operations/           # 运维文档
│   │       ├── monitoring-guide.md
│   │       ├── troubleshooting.md
│   │       └── backup-recovery.md
│   │
│   ├── audit/                    # 审计文档（流程层）
│   │   ├── 2026-04/             # 按年月组织

│   │   └── tools-reference.md   # 工具参考
│   │

├── CLAUDE.md                    # Claude Code项目配置（保留根目录）

```

### 文档命名规范
1. **英文命名优先**：主要文档使用英文命名，便于工具处理
2. **日期格式**：YYYY-MM-DD（2026-04-19）
3. **语义化命名**：`{domain}-{topic}-{date}.md`
4. **短横线分隔**：使用短横线而非下划线

## 迁移策略

### 分阶段迁移
**阶段1：建立基础结构**（1天）
- 创建docs目录和子目录结构
- 迁移核心架构文档（COGNITIVE_DNA.md, AGENTS.md）
- 更新README.md指向新文档结构
- 创建文档索引文件

**阶段2：迁移技术文档**（2天）
- 迁移部署指南和操作文档
- 重组技术规范文档
- 建立技术文档索引

**阶段3：归档审计报告**（2天）
- 按年月组织审计报告
- 创建审计汇总索引
- 删除重复或过时报告

**阶段4：完善用户文档**（1天）
- 填充Claude Code模板文件
- 创建用户指南和快速开始
- 更新工具参考文档

**阶段5：验证与清理**（1天）
- 验证所有链接和引用
- 清理根目录冗余文档
- 更新脚本中的文档路径引用

### 保留原位置文档
以下文档保留在根目录：
- `CLAUDE.md` - Claude Code项目配置
- `README.md` - 项目主README
- `.gitignore`等配置文件

## 质量门禁

### 迁移前检查
- [ ] 所有源文档已备份
- [ ] 文档分类方案通过评审
- [ ] 迁移工具和脚本准备就绪

### 迁移中检查
- [ ] 文档内容完整性验证（无内容丢失）
- [ ] 内部链接和引用更新验证
- [ ] 命名规范一致性检查

### 迁移后检查
- [ ] 所有文档可访问性测试
- [ ] 搜索引擎（如grep）能定位迁移后文档
- [ ] 关键脚本中的文档路径引用更新
- [ ] 文档索引文件完整准确

## 风险管理

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 内容丢失 | 低 | 高 | 1. 迁移前完整备份<br>2. 使用版本控制追踪变更<br>3. 分阶段验证内容完整性 |
| 链接失效 | 中 | 中 | 1. 自动化链接检查和更新<br>2. 创建重定向文件（如旧位置README.md）<br>3. 更新已知的脚本引用 |
| 分类错误 | 中 | 低 | 1. 制定明确的分类规则<br>2. 创建分类决策树<br>3. 允许后期调整分类 |
| 用户混淆 | 低 | 中 | 1. 保留根目录README指引<br>2. 创建文档地图<br>3. 更新项目入口点说明 |

## 成功标准

### 技术成功标准
1. 文档结构清晰，符合三才六层模型
2. 命名规范一致性>95%
3. 无内容丢失或损坏
4. 所有内部链接功能正常

### 操作成功标准
1. 文档查找时间减少50%以上
2. 新贡献者能快速找到所需文档
3. 文档维护工作量减少30%

### 项目成功标准
1. 文档成为项目知识管理的有效工具
2. 支持项目长期维护和团队协作
3. 为其他项目建立文档管理最佳实践
